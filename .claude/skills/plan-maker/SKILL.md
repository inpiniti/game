# Plan Maker — 요구사항 규모 분류 → 설계/플랜 문서

요청을 받으면 **바로 코드를 만지지 말고**, 먼저 규모를 분류하고 그 규모에 맞는 문서를 만든다.
이 프로젝트(딸기 서바이벌)는 "기준 문서는 얼리고, 바뀌는 것만 새 문서에 적는" 델타 방식으로 문서를 관리한다.

## 0. 분류 — 대 / 중 / 소

| 규모 | 기준 | 예시 (이 프로젝트 실사례) |
|---|---|---|
| **대** | **설계 자체가 바뀐다** — DB 스키마, 아키텍처/FSD 구조, 게임↔앱 계약, 화면 구조(라우트 추가·화면 신설), 횡단 관심사(전 화면의 전제가 바뀜) | i18n 도입(전 화면 문자열 전제 변경), 멀티플레이, 새 게임 모드 |
| **중** | 설계는 그대로, **여러 파일·여러 단계에 걸친 기능** — 기존 화면/모듈 안에서의 기능 추가, 2단계 이상으로 쪼개야 하는 작업 | 문제집 즐겨찾기, 결과 화면 공유 기능, 새 랭킹 탭 |
| **소** | 파일 몇 개로 끝나는 것 — 버그픽스, 문구/스타일 수정, 단일 컴포넌트 수정, 설정값 변경 | 버튼 색 변경, 오타 수정, 널 체크 추가 |

분류가 애매하면(중↔대 경계 등) 판단 근거와 함께 사용자에게 물어본다. 명확하면 **분류 결과와 근거를 한 줄로 말하고 바로 진행**한다.

## 1. "소" — features 경량 플랜 후 바로 작업

설계는 건드리지 않는다. `docs/features/YYYY-MM-DD_<슬러그>.md`에 **경량 플랜**을 만들고, 승인 대기 없이 바로 작업을 진행한다 (소는 플랜이 기록 목적이므로 멈추면 오히려 번거롭다).
소라도 모델 지정·에이전트 위임 규칙(4절)은 동일하다 — 보통 Haiku(low)나 Sonnet(medium) 에이전트 1개면 충분하다.

```markdown
# 기능: <이름> — <한 줄 요약>
> 작성일 · 분류: **소형** — <근거 한 줄>
## 왜
## 작업            ← 플랜 표 1~3행 (아래 "플랜 표 작성 규칙" 포맷, 단계 쪼개기 불필요)
## 완료 기준
```

## 2. "중" — features 델타 문서

설계(docs/design, docs/screen)는 건드리지 않는다. `docs/features/YYYY-MM-DD_<슬러그>.md` 하나를 만든다.
포맷은 기존 예시 [docs/features/2026-07-08_i18n.md](../../../docs/features/2026-07-08_i18n.md)를 따른다:

```markdown
# 기능: <이름> — <한 줄 요약>
> 작성일 · 기준 설계 문서 링크 · 분류: **중형**
## 1. 왜
## 2. 핵심 설계 결정        ← 개념 정의, 정책, 계약. 표/코드로 구체적으로
## 3. 화면 변경 (델타)      ← 레이아웃 변경 없으면 "조각 추가"만 ASCII로
## 4. 플랜 — 단계 × 스킬 × 모델   ← 아래 "플랜 표 작성 규칙" 참조
## 5. 하지 않는 것 (명시적 범위 제외)   ← ❌ 목록, 이유 포함
## 6. 리스크                ← 리스크 | 대응 표
```

## 3. "대" — 설계 버전업 + development 플랜

두 가지를 만든다. **버전 번호는 하드코딩하지 말고 디렉터리를 나열해서 최신 버전을 찾는다.**

1. **설계 버전업**: `docs/design/`의 최신 `supabase-quiz-fsd-vN.md`를 복사해 `v{N+1}`을 만들고 바뀌는 절을 수정·추가한다 (구버전은 얼림 — 수정 금지). 화면 구조(라우트·화면 신설)가 바뀌면 `docs/screen/screens-vM.md`도 같은 방식으로 `v{M+1}` 생성. 화면이 안 바뀌면 screen은 건드리지 않는다.
2. **플랜**: `docs/development/YYYY-MM-DD_<슬러그>-plan.md` — 포맷은 [docs/development/build-plan-skills-models.md](../../../docs/development/build-plan-skills-models.md)를 따른다 (사용법 / 모델 선택 기준 / 단계별 지정 표 / 스킬별 역할 요약 / 다음 단계).

## 4. 플랜 표 작성 규칙 (소·중·대 공통)

플랜의 핵심은 이 표다. 열은 반드시 다섯 개 — 모델 열에는 **effort·thinking까지 병기**한다:

| 단계 | 작업 | 사용 스킬 | 에이전트 모델 (effort) | 완료 기준 |
|---|---|---|---|---|
| i-0 기반 | … | codebase-design | **Opus** (high, thinking O) | … |
| i-1 추출 | … | i18n-expert | Sonnet (medium, thinking O) | … |
| i-2 시드 | … | — | Haiku (low, thinking X) | … |

- **단계**: 의존 순서를 명시한다 (`0 → 1 → (2 ∥ 3) → 4` 식으로 병렬 가능 구간 표시). 물량이 큰 단계는 파일 단위로 쪼개 병렬 에이전트 가능 여부를 적는다
- **사용 스킬**: **find-skills 스킬을 실제로 실행**해서 단계별 스킬을 정한다 —
  1. 먼저 이미 설치된 스킬에서 찾는다: 프로젝트(`.agents/skills/`, `.claude/skills/`)와 세션 전역 스킬 (build-plan-skills-models.md의 "스킬별 역할 요약" 표가 기존 스킬 카탈로그)
  2. 없으면 `npx skills find <키워드>`로 검색 — 1K+ 설치·신뢰할 수 있는 소스만 채택, 설치 후 플랜 문서에 출처를 남긴다 (예: `daymade/claude-code-skills@i18n-expert`)
  3. 맞는 스킬이 없는 단계는 억지로 채우지 말고 비워둔다
- **에이전트 모델 (effort)**: 모델 3티어 × effort로 지정한다. effort는 에이전트를 띄울 때 실제 파라미터로 전달되는 값(`low`/`medium`/`high`/`xhigh`/`max`)이고, thinking(추론) 깊이를 결정하므로 "thinking O/X"는 effort와 일관되게 적는다 (low=thinking 거의 X, high 이상=깊은 thinking O). 실행 시 Agent 도구의 `model`·`effort` 파라미터에 표의 값을 그대로 넣는다.

| 모델 | effort | 언제 |
|---|---|---|
| **Opus** | high~max, thinking O | 설계·계약·보안·복잡한 상태 로직 — 틀리면 비싼 것. 계약 하나에 전체가 걸리면 xhigh/max |
| **Sonnet** | medium(기본)~high, thinking O | 일반 기능 구현·UI·연동 — 주력. 통합·엣지케이스 많으면 high |
| **Haiku** | low~medium, thinking X | 기계적·반복·정적 — 판단 여지 적은 것 (시드 데이터, 보일러플레이트, 단순 치환) |

  - effort를 올리는 건 공짜가 아니다(시간·토큰). "틀리면 재작업 비용이 큰가?"로 판단하고, 기본값은 Sonnet(medium)
  - 검증·리뷰 단계는 실행은 Sonnet(medium)로 하되, 막히는 진단은 한 단계 위 모델/effort로 승급한다고 표에 명시
  - **"직접(에이전트 불필요)" 금지** — 모든 행에 모델·effort를 반드시 지정한다. 플랜을 짜는 세션은 보통 비싼 모델이라, 구현까지 직접 하면 Haiku로 충분한 일에 최고가 모델을 쓰는 낭비가 된다. 아무리 작은 작업(설정 파일 1개, 오타 수정)도 표의 모델로 에이전트를 띄워 위임한다

## 4-1. 실행도 표의 모델로 (플랜 = 실행 계약)

플랜 표의 모델 지정은 문서용 장식이 아니라 **실행 계약**이다. 구현 단계에 들어가면:

- 각 단계는 Agent 도구에 `model`·`effort`를 표의 값 그대로 넣어 서브에이전트로 실행한다
- 플랜을 짠 현 세션은 **오케스트레이션만** 한다 — 분류·플랜 작성·에이전트 결과 검수·병합·사용자 보고. 코드 구현을 직접 하지 않는다
- 예외는 승급뿐: 에이전트가 막히거나 결과가 반복 불합격이면 한 단계 위 모델/effort로 재시도하고, 그 사실을 플랜 문서에 기록한다

- **완료 기준**: 각 단계가 끝났음을 확인할 수 있는 관찰 가능한 조건 한 줄 (예: "ja로 플레이 시 HUD가 일본어")

## 5. 플랜 후 멈춘다 (중·대)

중·대는 **문서 작성까지가 이 스킬의 일이다.** 문서를 만들고 요약을 보여준 뒤, 사용자가 "플랜에 따라 개발해줘" 등으로 승인할 때까지 구현을 시작하지 않는다. 플랜 단계에서 사용자와 조율한 변경은 문서에 반영한 뒤 진행한다.
소는 예외 — 경량 플랜을 만든 즉시 작업까지 진행한다 (1절).
