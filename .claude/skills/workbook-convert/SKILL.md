# Workbook Convert — 원본 단어장 → 앱 업로드용 CSV/JSON

## 이 스킬이 푸는 문제

`docs/workbook/`에는 출처가 제각각인 단어장 원본이 쌓인다(.txt, .xls, .xlsx). 만든 사람이
달라서 **입력 형식은 파일마다 다르지만**, 앱 업로드 폼이 받는 **출력 형식은 하나로 고정**돼야
결과물이 들쭉날쭉하지 않다. 그래서 변환을 두 층으로 나눈다:

- **① 읽는 규칙 (파일마다 새로 파악)** — "이 파일은 `단어[탭]뜻`", "얜 `단어: 뜻1, 뜻2`",
  "엑셀은 A열=단어·C열=뜻" 같은 것. 재사용 안 됨, 매번 파일을 실제로 열어 확인한다.
- **② 내보내는 규칙 (전 파일 공통, 한 번 정하면 고정)** — 아래 "출력 계약"과 "정제 규칙".
  첫 파일에서 사용자와 합의한 뒤 나머지 파일에 **똑같이** 적용한다.

목표: 원본이 어떻게 생겼든 최종 결과 10개가 **일관된 품질·동일 포맷**으로 나오게 하는 것.

## 실행 모델 (effort · thinking)

이 스킬을 서브에이전트로 돌릴 때 단계별 모델·effort를 아래대로 지정한다. 판단이 필요한 곳엔
thinking을 켜고, 규칙이 확정된 뒤 기계적으로 찍어내는 곳은 낮춰서 시간·토큰을 아낀다.

| 단계 | 작업 성격 | 모델 (effort · thinking) |
|---|---|---|
| **①+② 형식 파악 + 규칙 확정** (첫 파일, 흐름 1~2) | 지저분한 원본에서 열 구조·구분자 추론, 정제 방침 판단 — 틀리면 배치 전체가 오염 | **Sonnet (medium, thinking O)** |
| **③④ 전체·나머지 변환** (규칙 확정 후, 흐름 3~4) | 확정 규칙을 기계적으로 적용해 CSV 생성 — 판단 여지 적음 | **Haiku (low, thinking X)** |
| 예외: 원본이 심하게 불규칙 (병합 셀·다열 혼재·인코딩 깨짐) | 파싱 자체가 판단을 요구 | Haiku로 막히면 **Sonnet (medium~high)** 로 승급 |

- 기본값은 "규칙 잡을 땐 Sonnet, 찍어낼 땐 Haiku". 한 파일 안에서도 1~2단계는 Sonnet, 3단계는 Haiku로 나눠 위임할 수 있다.
- 파일이 단순한 `단어[탭]뜻` txt 하나뿐이면 전 과정을 Haiku(low)로 끝내도 된다 — effort를 올리는 건 공짜가 아니다.
- 막히는 진단(왜 열이 안 잡히나, 왜 인코딩이 깨지나)은 한 단계 위 모델/effort로 올려 재시도한다.

## 출력 계약 (앱이 실제로 받는 형식 — 절대 어기지 말 것)

앱 파서(`src/shared/lib/quizParser.ts`)가 그대로 받는 형식이다. 여기서 벗어나면 업로드 미리보기에서
행 오류가 난다.

**CSV (기본 권장)** — 열 순서 `front,back,example`:
```csv
front,back,example
apple,사과,I ate an apple.
run,달리다;뛰다,
anywhere,어딘가에;아무데도;어디든지,You can sit anywhere.
```
- **헤더 행 `front,back,example`을 첫 줄에 넣는다** (파서가 자동 인식해 건너뜀 — 사람이 열어봐도 명확).
- `front` = 학습 언어 단어(영어 등), `back` = 뜻(모국어). 문제/답 문제집이면 front=문제, back=답.
- **한 단어의 여러 뜻은 `;`로 잇는다** (`달리다;뛰다`). 첫 항목이 대표 표기. 쉼표 아님 — 쉼표는 열 구분자다.
- `example`(예문)은 없으면 빈칸. 맨 끝 열이라 값 없으면 그냥 콤마 뒤를 비운다.
- **쉼표·줄바꿈·따옴표가 들어간 필드는 반드시 RFC4180 따옴표 처리** — 손으로 하지 말고
  Python 표준 `csv` 모듈(`csv.writer`)로 써라. 그러면 이스케이프가 자동으로 맞는다. (아래 "쓰기" 참조)

**JSON (원하면)** — 최상위 배열, 각 원소 `{ "front", "back", "example"? }`:
```json
[{ "front": "apple", "back": "사과", "example": "I ate an apple." },
 { "front": "run", "back": "달리다;뛰다" }]
```

CSV·JSON 어느 쪽이든 앱은 받는다. 대용량 목록은 CSV가 가볍고 사람이 검수하기 쉬워 기본값으로 쓴다.

## 정제 규칙 (② — 첫 파일에서 확정 후 전 파일 공통)

원본에는 앱에 넣기엔 지저분한 것들이 섞여 있다. 아래는 **기본 정제 방침**이다. 첫 파일 샘플을
보여줄 때 이 방침을 명시하고, 사용자가 바꾸고 싶어 하면 조정한 뒤 **그 결정을 나머지 파일에 그대로 적용**한다.

- **품사 표기**(`n.`, `v.`, `adj.`, `(명)`, `(동)`): 기본 **제거**. 뜻만 남긴다.
- **발음기호/IPA**(`[əˈpɪr]`, `/rʌn/`): 기본 **제거**.
- **뜻 개수**: 원본에 뜻이 여럿이면 **최대 3개까지** `;`로 잇고 나머지는 버린다(게임 주관식 채점 부담↓).
  너무 많은 뜻은 오히려 방해 — 대표적인 것만.
- **번호/불릿**(`1.`, `①`, `- `): 제거하고 뜻만.
- **예문**: 원본에 있으면 `example`로 옮기고, 없으면 만들지 않는다(빈칸). 지어내지 말 것.
- **중복 표제어**: 같은 front가 여러 줄이면 뜻을 `;`로 합쳐 한 줄로.
- **빈 행·머리말·페이지 번호·저작권 줄** 같은 잡음은 버린다.
- 원본 뜻 안에 이미 `,`로 여러 뜻을 나열했으면 → `;`로 바꾼다(쉼표는 CSV 열 구분자라 충돌).

이 규칙들은 출발점일 뿐이다. 파일 성격(속담·숙어는 뜻이 길고 문장형, GRE는 어려운 1:1 대응)에
따라 사용자와 조정하되, **한 번 정한 규칙은 배치 전체에 일관되게** 간다.

## 출력 위치·파일명

- 저장 폴더: **`docs/workbook/conversion/`** (없으면 만든다).
- 파일명: **`{제목}_{대상}_{언어}.csv`** — 이 세 조각은 사용자가 앱 업로드 폼에 그대로 입력할
  메타데이터다(파일명이 곧 메모).
  - **제목**: 원본을 대표하는 이름 (예: `중학교 영단어 1200`, `SAT 5000`).
  - **대상**: `초등` · `중등` · `고등` · `일반인` 중 하나 (앱 카테고리와 동일). 파일 내용·제목으로
    추정하되 애매하면 사용자에게 묻는다 (예: "중학교 1200"→중등, "SAT/GRE/토익"→일반인, "수능"→고등).
  - **언어**: 학습 언어 코드 `en`(영어) · `ja`(일본어) · `zh`(중국어). 대부분 `en`.
  - 예: `중학교 영단어 1200_중등_en.csv`, `SAT 5000_일반인_en.csv`.

## 작업 흐름

### 1. 원본 열어서 형식 파악 (①)
- `.txt`: Read로 앞부분 수십 줄을 본다. 구분자(탭·콜론·여러 칸 공백·줄 패턴)와 열 의미를 파악.
- `.xls`/`.xlsx`: 바이너리라 그냥 못 읽는다. **Python으로 덤프**해서 열 구조를 본다:
  ```bash
  python -c "import pandas as pd; df=pd.read_excel(r'경로', header=None, dtype=str); print(df.head(20).to_string())"
  ```
  `pandas`가 없으면 `pip install pandas openpyxl xlrd` (xls는 `xlrd`, xlsx는 `openpyxl`). 어느 열이
  단어/뜻/예문인지, 헤더 행이 몇 줄인지 확인한다.

### 2. 첫 파일로 규칙 확정 (② 합의)
- 파악한 형식으로 앞 10~20개를 변환한 **샘플을 보여준다**. 이때 적용한 정제 규칙(품사 제거, 뜻 3개 제한 등)을
  한 줄로 밝힌다.
- 사용자가 규칙을 조정하면 반영하고, **그 규칙을 이 배치의 확정 규칙으로 삼는다.**
- 대상·언어가 애매하면 여기서 확인한다.

### 3. 전체 변환 + 쓰기
- 확정 규칙으로 파일 전체를 변환한다. **CSV 쓰기는 Python `csv` 모듈로** — 이스케이프를 손으로 만들지 말 것:
  ```python
  import csv
  rows = [("apple","사과","I ate an apple."), ("run","달리다;뛰다","")]  # (front, back, example)
  with open(r'docs/workbook/conversion/제목_대상_언어.csv', 'w', encoding='utf-8-sig', newline='') as f:
      w = csv.writer(f)
      w.writerow(["front","back","example"])
      w.writerows(rows)
  ```
  `utf-8-sig`(BOM)로 저장하면 한글이 엑셀에서도 안 깨진다. 앱 파서는 BOM을 무시하므로 안전.
- 변환 후 **행 수와 눈에 띄는 이상(빈 뜻, 깨진 인코딩)** 을 한 번 점검해 보고한다.

### 4. 나머지 파일 (①만 새로, ②는 고정)
- 각 파일은 형식 파악만 새로 하고, 정제·출력 규칙은 확정된 것을 그대로 적용한다.
- 여러 파일이면 파일마다 대상·언어를 정하고(제목/내용 기반, 애매하면 질문) 같은 방식으로 변환.
- 특정 파일이 확정 규칙에 안 맞을 만큼 특이하면(예: 속담은 뜻이 문장 하나뿐) 예외를 사용자에게 짚어 확인한다.

## 하지 않는 것
- **예문·뜻을 지어내지 않는다.** 원본에 없으면 빈칸으로 둔다. 없는 걸 채우면 학습 데이터가 오염된다.
- **DB에 직접 넣지 않는다.** 이 스킬은 파일 변환까지만 — 업로드는 사용자가 앱 `/sets` 폼에서 한다
  (실서비스 데이터라 사람이 미리보기로 확인 후 올리는 게 안전).
- 원본 파일을 수정하거나 옮기지 않는다. 결과만 `conversion/`에 새로 쓴다.
